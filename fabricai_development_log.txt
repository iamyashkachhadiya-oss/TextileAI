STAGE 6: BINDING & ERROR RESOLUTION
--------------------------------------------------------------------------------
[ERRORS ENCOUNTERED & RESPONSES]

ERROR INSTANCE #1: Next.js Component Hydration with Canvas API
- Incident: Realized HTML Canvas node references might conflict during Server Side Rendering pipelines depending on React's Hydration tree.
- Resolution: Strictly decoupled Canvas interaction and specifically enforced `'use client'` tags atop active component trees, bounding instantiation loops strictly into `useEffect` cycles ensuring the `window` construct was explicitly available.

ERROR INSTANCE #2: TypeScript 5 Explicit Casting (TS2352)
- Incident: Encountered compiling error inside `src/components/design/LoomForm.tsx` via `(loom as Record<string, unknown>)[key] as number`. TypeScript warned the object overlap lacked sufficient narrowing context.
- Resolution: Restructured the type coercion chain to pass cleanly through an unknown bridge: `(loom as unknown as Record<string, number>)[key]`. The compiler immediately validated the assertion without mutability regressions.

ERROR INSTANCE #3: Automated Subagent Capacity Depletion (API 503)
- Incident: The internal structural testing routine running a browser subagent met a capacity limit (UNAVAILABLE code 503 HTTP status) while executing the complex `/demo` end-to-end navigational tour.
- Resolution: Circumvented the blocked subagent pipeline. Relied on manual macro compilation verifications using `npm run build` and `npx tsc --noEmit` yielding zero remaining issues, then deployed `demo/page.tsx` utilizing a placeholder hardcoded data injection allowing immediate user verification without Supabase configurations.

STAGE 8: ADVANCED WEFT & NOZZLE SYSTEM
--------------------------------------------------------------------------------
[PROMPT / BUSINESS REQUIREMENT]
Transition the application from a static two-yarn model into a professional, unlimited-yarn "Multi-Weft" environment. Requirements:
1. Support for infinite distinct weft yarn types.
2. Tracking of engineering-grade physical properties (Shrinkage Min/Max, Tensile Strength cN, Elasticity %, Moisture Regain %, Dye Affinity).
3. Advanced Nozzle Control: Sequential assignment of loom nozzles (1, 2, 3...) per yarn.
4. Master Insertion Patterns: A user-defined repeat sequence of yarns to determine fabric weight and appearance.

[STEP-BY-STEP ACTIONS]
STEP 8.1: Data Model Migration (`types/index.ts`)
- Deprecated `weftA`/`weftB` in favor of a dynamic `WeftSystem` container.
- Engineered-in `NozzleConfig` and `YarnProperties` interfaces to hold high-fidelity industrial metadata.
- Created `InsertionSequence` to track the master pick pattern.

STEP 8.2: Multi-Yarn Calculation Engine (`engine.ts`)
- Re-architected `runAllCalculations` to handle weighted averages.
- Implemented `per_yarn_weft_weights` (g/100m) tracking, providing individual consumption metrics for every material in the weave.
- Added logic for `total_nozzles_in_use` and `effective_ppi` across advanced insertion patterns.

STEP 8.3: Nozzle-Aware Machine Exports (`machineFile.ts`)
- Overhauled `.EP`, `.WEA`, `.JC5`, and `.DES` generators.
- Injected specific nozzle/color selection bytes (`[C1]`, `Nozzle=2`, etc.) into every pick line, derived dynamically from the insertion sequence.

STEP 8.4: Dynamic Weft UI (`WeftForm.tsx`)
- Built the `WeftYarnSystem` component featuring:
  - Simple/Advanced mode toggling.
  - Unlimited "Yarn Cards" with collapsible engineering details.
  - A visual "Master Pattern Builder" using interactive chips.
  - Real-time nozzle sequence visualization.
